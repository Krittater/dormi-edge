ทำไมต้อง "build migration runner" แยก?

เพราะ image ที่รันจริง (runtime) รัน migration ไม่ได้ — มันติดตั้งแบบ --omit=dev (ไม่มี ts-node) และ copy มาแค่ dist/ (ไม่มี src/) ส่วนคำสั่ง migration:run ต้องใช้ทั้งสองอย่าง. ทางแก้: docker build --target build ดึง stage build ที่มีของครบ มารันเฉพาะกิจ แล้วลบทิ้ง — ไม่ต้องแตะ backend repo เลย.

🛠️ ผลรีวิว

Bug ที่เจอและแก้แล้ว

1. set -e ตัด error handler ตอน pg_dump พัง แก้แล้ว

เดิมเขียน docker exec ... pg_dump > file ลอยๆ — ถ้า pg_dump คืนค่า error, set -e จะ exit ทันที ข้ามการ reset git ทิ้งสถานะค้าง. แก้โดยห่อ if ! ...; then ให้ handler ทำงานก่อน:

if ! docker exec dormi_postgres sh -c 'pg_dump ... -Fc' > "$BACKUP_FILE"; then
    # ❌ pg_dump พัง → ลบไฟล์ว่าง + git reset + exit
fi

2. set -e ตัด handler ตอน build migration runner พัง แก้แล้ว

ถ้า code ใหม่ compile ไม่ผ่าน → docker build พัง → เดิม exit ทันทีทิ้ง git ค้างที่ commit ใหม่. แก้โดยห่อ if ! เช่นกัน — build พังก็ คืน code เดิม ก่อน exit.

จุดที่ตรวจแล้วถูกต้อง ยืนยัน

  • backup ไม่ต้องรู้รหัสผ่าน — pg_dump -U "$POSTGRES_USER" ดึง env จากในตัว container เอง (single-quote ให้ขยายฝั่ง container ไม่ใช่ฝั่ง host)
  • -e override --env-file — บังคับ DATABASE_HOST=postgres, PORT=5432 (ค่าเชื่อมภายใน container) ทับค่าใน env file ได้ถูกต้อง
  • ไม่ชนกับ scheduler — scheduler ยัง migrationsRun=true ตอน boot แต่หลัง explicit migrate จะไม่เจอ pending → no-op (TypeORM เช็คตาราง migrations ก่อน)
  • rotate ถูก — ls -1t | tail -n +8 เก็บ 7 ไฟล์ใหม่สุด ลบที่เหลือ
  • ลำดับปลอดภัย — backup → migrate เกิด ก่อน up -d เสมอ, postgres (persistent) ไม่ถูกแตะ

⚠️ ข้อจำกัดที่ควรรู้ (ไม่ใช่ bug — ธรรมชาติของ migration)

1. scheduler ยัง migrate เองตอน boot (fallback ไม่มี backup)

ถ้ามี migration ค้างจากรอบก่อนที่ deploy.sh ไม่ทัน guard, scheduler จะ migrate ให้ตอน boot โดยไม่มี backup. ถ้าอยากให้ deploy.sh เป็นทางเดียวจริงๆ ต้องปิด migrationsRun ใน backend (คนละ repo — ทำทีหลังได้).

2. TypeORM ห่อ transaction ให้ "เกือบ" ทุกกรณี

default Postgres รัน migration ใน transaction (พังแล้ว rollback อัตโนมัติ) — ยกเว้น migration ที่ตั้ง transaction = false เอง หรือใช้ CREATE INDEX CONCURRENTLY. กรณีพวกนี้ backup คือตัวกันพังจริง (จึงต้อง backup ก่อนเสมอ).

3. rollback code หลัง migrate แล้ว = schema ล้ำหน้า

ถ้า app พังหลัง migrate สำเร็จ แล้วต้อง rollback code เก่า → DB จะมี schema ใหม่กว่า code. migration แบบเพิ่ม (additive) code เก่ามักยังทำงานได้; แบบลบ/แก้ (destructive) อาจต้อง restore DB จาก backup. script พิมพ์คำสั่ง pg_restore ให้พร้อมในตอน rollback.

💡 แนวปฏิบัติ: เขียน migration แบบ additive (expand → contract) จะปลอดภัยกับ rolling deploy ที่สุด

📋 ขั้นตอนถัดไป

  • Push เอง (ตามกฎ) — แก้ที่ local เสร็จแล้ว, ยังไม่ได้ push ให้
  • ทดสอบตอน push: commit ธรรมดาต้องขึ้น ✅ ไม่มี migration ใหม่; commit ที่มี migration ต้องเห็น log 📦 → 🏗️ → 🚀 → ✅ และมีไฟล์ใน /root/db-backups/
  • อยากให้แน่นขึ้น (ทำทีหลัง): ปิด migrationsRun ใน backend ให้ deploy.sh เป็นทางเดียว · เพิ่ม health-check /whoami แบบลึกในบล็อกตรวจหลัง deploy
